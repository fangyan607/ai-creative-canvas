# Milestone v0.2: MVP

**Status:** ✅ SHIPPED 2026-07-02
**Phases:** 1-8
**Total Plans:** 34

## Overview

Complete MVP delivery of AI无限创意画布 (AI Unlimited Creative Canvas). From bare monorepo to fully integrated creative tool with infinite canvas, visual node editor, DAG execution engine, multi-provider AI generation, and polished application UI — all covered by 418 tests.

## Phases

### Phase 1: Core Canvas

**Goal**: Users have a performant, persistent infinite canvas with full element management
**Depends on**: Nothing (first phase)
**Plans**: 6 plans

Plans:

- [x] 01-01: Monorepo foundation — pnpm workspace, vendored Excalidraw fork, shared types, Vite 8 SPA skeleton
- [x] 01-02: Stores & AIElement — CanvasStore, HistoryStore, AIElement type in Excalidraw, store stubs, tests
- [x] 01-03: Canvas bridge — CanvasWrapper (Excalidraw + Zustand), App integration, keyboard undo/redo
- [x] 01-04: Layer panel — CanvasStore layer ordering, LayerPanel with reorder/lock/hide/group
- [x] 01-05: Performance — chunk rendering (2000x2000px), resolution-tiered AIElement, LRU image cache (200MB)
- [x] 01-06: Persistence — Dexie.js IndexedDB, project save/load, auto-save (180ms), Save/Save As UI

**Details:**
- Forked Excalidraw v0.18.0 (MIT) — minimal invasive changes tracked in FORK_CHANGES.md
- 5 Zustand stores: CanvasStore, HistoryStore, UIPrefsStore, EngineStore, AIQueueStore
- Chunk rendering system: 2000x2000px tiles, distance-based resolution tiers
- LRU image cache: 200MB cap with eviction, SHA-256 content hashing
- IndexedDB via Dexie.js: single-table schema, 180ms debounced auto-save

### Phase 3: Node Engine

**Goal**: Node graphs execute in correct topological order with incremental updates and organizational structure
**Depends on**: Phase 2
**Plans**: 5 plans

Plans:

- [x] 03-01: Types & Contracts — parentId, GroupNodeData, ExecutionStatus, engine type system
- [x] 03-02: Engine Core — toExecutionLayers, findAffectedDownstream, NodeEngine, EngineStore, stub resolvers, unit tests
- [x] 03-03: GroupNode + Status UI — GroupNode with collapse/expand, BaseNode status indicator
- [x] 03-04: Store Extensions — NodeGraphStore group CRUD, HistoryStore engine state
- [x] 03-05: Wiring + Verification — useAutoExecute, NodeEditorOverlay integration, App.tsx

**Details:**
- toExecutionLayers: topological sort producing ordered execution layers for parallel processing
- findAffectedDownstream: dirty-path marking — only re-executes nodes on the changed dependency chain
- NodeEngine: orchestrates execution layers, manages execution status, aborts on disconnected graphs
- EngineStore: tracks node statuses (idle/queued/processing/done/error), execution log, progress
- GroupNode: collapsible sub-graph container with named groups, expand/collapse toggle
- useAutoExecute: 180ms debounced auto-execution on any node graph change

### Phase 4: AI Adapters

**Goal**: System generates images through multiple AI providers via a clean adapter pattern with user-owned keys
**Depends on**: Nothing (can be developed in parallel with Phases 2-3)
**Plans**: 6 plans

Plans:

- [x] 04-01: Adapter Interface + Mock — AiAdapter abstract class, MockAdapter returning test images
- [x] 04-02: Adapter Registry + Plugin — AdapterRegistry with dynamic registration
- [x] 04-03: OpenAI DALL-E 3 — OpenAiAdapter implementing text-to-image
- [x] 04-04: Stability.ai — StabilityAdapter with text-to-image and image-to-image
- [x] 04-05: BYOK + ProviderStore — AES-256-GCM encrypted key storage, ProviderStore with Dexie persistence
- [x] 04-06: Prompt Builder — Tempura-based template engine, 7 templates across providers, 4-source context merging

**Details:**
- AiAdapter abstract class: interface-driven design, extendable without modifying existing code
- MockAdapter: returns SVG-based test images, zero API calls, instant response
- OpenAiAdapter: DALL-E 3 support with quality/style parameters, error handling, timeout
- StabilityAdapter: text-to-image and image-to-image, generation parameter mapping
- AdapterRegistry: Map-based dynamic registration, provider discovery by capability
- ProviderStore: Dexie-backed key storage with AES-256-GCM encryption (Web Crypto API)
- PromptBuilder: Tempura template engine (1.3KB gzip), Handlebars-compatible {{variable}} syntax
- 7 centralized templates: mock, openai (default/fast/quality), stability (default/artistic/photo-realistic)
- 4-source priority merging: system < global < upstream < params

## Milestone Summary

**Key Decisions:**
- Excalidraw v0.18.0 fork (MIT) — minimal invasive changes, FORK_CHANGES.md tracking
- 5 domain-split Zustand stores (not monolithic) — CanvasStore, HistoryStore, UIPrefsStore, EngineStore, AIQueueStore
- React 19 + Vite 8 + pnpm 11 + Node 24 tech stack — latest 2026 ecosystem
- SSE streaming (not WebSocket) for AI progress — simpler, sufficient for one-directional updates
- Adapter pattern for AI providers — built before any real provider integration
- Tempura (1.3KB) over Handlebars (82KB) for prompt templates — same syntax, 63x smaller
- AES-256-GCM for API key encryption via Web Crypto API
- Phase execution order: 1 → 2 → 3 → (4 parallel 6) → 5 → 7 → 8

**Issues Resolved:**
- Excalidraw merge conflicts during fork integration — resolved via minimal patch approach
- Cross-store synchronization between canvas and node graph undo/redo
- Vite proxy configuration for SSE streaming
- Chrome extension cross-origin errors with SSE
- Code review findings across all phases (CR, WR issues addressed)
- Module resolution issues with TypeScript subpath exports in ai-core

**Issues Deferred:**
- CI Pipeline — local run only (D-08 deferred decision)
- E2E flow variants (template, settings, dark mode) — core flow only
- Full performance suite (load test, frame rate benchmark)
- Pre-commit hooks (husky + lint-staged)
- Excalidraw SCSS deprecation warnings (upstream code)
- useAutoSave.ts dead code cleanup (replaced by inline useProjectAutoSave)
- E2E spec hardening: .catch(() => false), vacuous assertion, hardcoded timeouts

**Technical Debt Incurred:**
- `useAutoSave.ts` dead code — no longer imported by production code
- E2E spec: `.catch(() => false)` silently swallows exceptions on prompt textarea
- E2E spec: vacuous assertion `.toBeGreaterThanOrEqual(0)` always passes
- E2E spec: hardcoded `waitForTimeout` instead of UI-based waits
- `resolvers.test.ts`: async resolvers only tested for Promise type, not return values
- 4 phases missing Nyquist VALIDATION.md compliance
