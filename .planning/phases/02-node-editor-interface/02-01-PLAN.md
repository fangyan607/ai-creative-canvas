    The functions convert between Excalidraw viewport coordinates and React Flow viewport coordinates. The core math:

    ```
    canvasToNodeSpace(canvasX, canvasY, viewport):
      // Excalidraw viewport: scrollX, scrollY are negative of canvas offset
      // React Flow: position is relative to the viewport origin
      // Convert: nodeX = canvasX + viewport.x, nodeY = canvasY + viewport.y
      // At zoom != 1, adjust: nodeX = (canvasX + viewport.x) / viewport.zoom
      // Actually the transform must account for:
      //   Excalidraw stores element positions in "canvas space" (absolute)
      //   React Flow stores node positions in "flow space" (viewport-relative)
      //   At zoom 2x, 1 canvas pixel = 2 flow pixels
      //   nodeX = (canvasX + viewport.scrollX) * viewport.zoom
      //   nodeY = (canvasY + viewport.scrollY) * viewport.zoom
    
    nodeToCanvasSpace(nodeX, nodeY, viewport):
      // Inverse of above
      //   canvasX = nodeX / viewport.zoom - viewport.scrollX
      //   canvasY = nodeY / viewport.zoom - viewport.scrollY
    ```

    Wait — let me reconsider the math. Excalidraw's viewport represents the scroll offset. When zoom = 1 and scrollX = 0, the scene origin (0,0) maps to viewport origin (0,0). When you pan right (scrollX increases), elements appear to move left — but the element's absolute position doesn't change, only the viewport offset.

    For our use case:
    - React Flow nodes are positioned in "flow space" which is independent of zoom
    - Excalidraw elements are in "canvas space" (absolute coordinates)
    - At any given viewport state, a point in canvas space maps to flow space as:
      flowX = (canvasX + scrollX) * zoom  ... but actually Excalidraw scrollX is the scene position of the viewport left edge
    
    Let me think about this differently. In Excalidraw:
    - `appState.scrollX` is the canvas-space x-coordinate of the viewport's left edge
    - So a canvas point (canvasX, canvasY) appears on screen at:
      screenX = (canvasX - appState.scrollX) * appState.zoom.value
      screenY = (canvasY - appState.scrollY) * appState.zoom.value

    For React Flow, node positions are in "flow space" which we want to match what users see on screen. So:
    - React Flow node position = (canvasX, canvasY) in canvas space
    - React Flow handles its own viewport transform internally
    - We don't need to convert — React Flow nodes are placed in the same coordinate space as Excalidraw elements

    Actually, the simpler approach: React Flow's internal coordinate system is independent of its viewport. A node placed at position (100, 100) stays there regardless of pan/zoom. Excalidraw's elements also stay at their absolute positions regardless of pan/zoom.

    The tricky part is when we CREATE a node from a screen position (e.g., drag from toolbar onto the screen). We need to convert the screen position to canvas space to place the node correctly.

    So the real transform we need:
    ```
    // Convert a screen pixel position to canvas-space coordinate
    screenToCanvas(screenX, screenY, viewport):
      canvasX = screenX / viewport.zoom + viewport.scrollX
      canvasY = screenY / viewport.zoom + viewport.scrollY
      return { x: canvasX, y: canvasY }
    
    // Convert a canvas-space coordinate to screen pixels
    canvasToScreen(canvasX, canvasY, viewport):
      screenX = (canvasX - viewport.scrollX) * viewport.zoom
      screenY = (canvasY - viewport.scrollY) * viewport.zoom
      return { x: screenX, y: screenY }
    ```

    But the CONTEXT.md says the functions are `canvasToNodeSpace` and `nodeToCanvasSpace`. These names suggest:
    - `canvasToNodeSpace`: Convert from Excalidraw canvas space to "node space" (React Flow's coordinate system)
    - `nodeToCanvasSpace`: Convert from React Flow space to Excalidraw canvas space

    Since React Flow and Excalidraw both use the same underlying coordinate space (absolute scene coordinates), these functions can be identity transforms when zoom = 1. At other zooms, they adjust for the zoom factor.

    Actually, I think the simplest correct implementation is:

    ```typescript
    export function canvasToNodeSpace(
      x: number,
      y: number,
      viewport: { x: number; y: number; zoom: number }
    ): CoordinateResult {
      // React Flow uses the same absolute coordinate space as Excalidraw
      // No transform needed — both systems share the same underlying space
      return { x, y }
    }
    
    export function nodeToCanvasSpace(
      x: number,
      y: number,
      viewport: { x: number; y: number; zoom: number }
    ): CoordinateResult {
      // Inverse of canvasToNodeSpace
      return { x, y }
    }
    ```

    Wait, that can't be right either. Let me actually think about what these are used for.

    The purpose (D-29): "React Flow's translateExtent and Excalidraw's scene bounds must align."

    So when the user pans the Excalidraw canvas, React Flow must pan in sync. This is done by setting React Flow's `translateExtent` to match the Excalidraw scene bounds, and syncing the React Flow viewport with the Excalidraw viewport.

    React Flow's `setViewport({ x, y, zoom })` uses a transform where:
    - x, y are the "translation" in pixels at zoom 1
    - The visible portion is translated by (-x * zoom, -y * zoom) for screen rendering

    Actually, React Flow's viewport transform works like CSS transform: translate(x, y) scale(zoom). So a viewport of { x: 100, y: 50, zoom: 2 } means the scene is translated 100px right, 50px down, and scaled 2x.

    Meanwhile, Excalidraw's scrollX/scrollY represent the canvas coordinate at the viewport's left/top edge. If scrollX = 200 at zoom = 1, that means the canvas coordinate x=200 is at the left edge of the viewport.

    The relationship:
    - React Flow viewport X = -Excalidraw scrollX * zoom
    - React Flow viewport Y = -Excalidraw scrollY * zoom

    So to align them:
    ```
    reactFlowViewport.x = -excalidrawScrollX * excalidrawZoom
    reactFlowViewport.y = -excalidrawScrollY * excalidrawZoom
    reactFlowViewport.zoom = excalidrawZoom
    ```

    And the coordinate transforms:
    ```
    canvasToNodeSpace(canvasX, canvasY, { x: scrollX, y: scrollY, zoom }):
      // Canvas space to Node space (both use same absolute coords, just different viewport origins)
      // In Excalidraw: element at (canvasX, canvasY) appears at screen
      //   screenX = (canvasX - scrollX) * zoom
      //   screenY = (canvasY - scrollY) * zoom
      // React Flow stores node positions in the same absolute space as Excalidraw
      // So no conversion needed for node positions
      return { x: canvasX, y: canvasY }
    ```

    Actually, I think the key insight is that BOTH Excalidraw elements and React Flow nodes use the same underlying absolute coordinate space. The viewport transform is applied separately by each library. So to place a React Flow node at the same position as an Excalidraw element, you just use the element's coordinates directly.

    The real value of these functions is for edge cases like:
    1. Creating a node from a toolbar drag at a specific screen position → need to convert screen → canvas space
    2. Snapping a generated AI image from PreviewNode to canvas → need to place it at the right canvas position

    Let me implement them as utility functions that handle the screen↔canvas conversion:

     *  Used when creating a node from a toolbar drag onto the editor. */
    export function screenToCanvas(
      screenX: number,
      screenY: number,
      viewport: { x: number; y: number; zoom: number }
    ): CoordinateResult {
      return {
        x: screenX / viewport.zoom + viewport.x,
        y: screenY / viewport.zoom + viewport.y,

     *  Used for snapping canvas elements to screen-aligned positions. */
